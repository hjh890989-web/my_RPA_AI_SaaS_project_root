**Scenario 1: 합의 테이블 정상 읽기 + 동기화 성공**
- **Given**: Mock ERP 테이블(재고: `t_inv_stock`, 발주: `t_pur_order`, 실적: `t_prd_result`)이 `approved_tables`에 등록되어 있음
- **When**: `POST /api/v1/erp/sync` (connection_id=C1, table_names=["t_inv_stock", "t_pur_order"])를 실행함
- **Then**: 해당 테이블의 데이터가 SELECT 쿼리로만 읽혀 FactoryAI DB에 동기화되고, 동기화 ≤5분 내 완료, `sync_status=COMPLETED`, `records_count`가 정확히 기록된다. **Write(INSERT/UPDATE/DELETE) 쿼리 실행 = 0건**.

**Scenario 2: Write(INSERT) 시도 → 3중 차단**
- **Given**: 악의적 또는 프로그래밍 오류로 ERP 테이블에 INSERT를 시도함
- **When**: Layer별 차단을 검증함
- **Then**:
  - **Layer 1 (TS)**: 컴파일 에러 발생 (Write 메서드 미정의)
  - **Layer 2 (Prisma MW)**: 런타임에서 INSERT 쿼리를 인터셉트하여 차단 + 로깅
  - **Layer 3 (RLS)**: DB 레벨에서 최종 차단
  - CISO에게 ≤10초 내 보안 알림이 발송됨

**Scenario 3: 미합의 테이블 접근 시도 → 즉시 차단**
- **Given**: `approved_tables`에 포함되지 않은 `t_hr_employee`(인사 테이블)에 접근을 시도함
- **When**: 동기화 요청에 해당 테이블을 포함함
- **Then**: 403 Forbidden이 즉시 반환되고, 감사 로그에 "미합의 테이블 접근 시도"가 기록된다.

**Scenario 4: UPDATE/DELETE 시도 → 동일한 3중 차단**
- **Given**: ERP 테이블에 UPDATE 또는 DELETE를 시도함
- **When**: 3개 Layer에서 순차 검증함
- **Then**: INSERT와 동일하게 모든 Write 유형이 3중 차단된다.
