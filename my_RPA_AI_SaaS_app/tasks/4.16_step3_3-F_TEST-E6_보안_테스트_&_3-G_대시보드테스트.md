### :test_tube: Acceptance Criteria (BDD/GWT)

**Scenario 1: RLS — OPERATOR가 자기 공장 데이터만 접근 가능**
- **Given**: OPERATOR 역할의 USER-A가 FACTORY-A에 소속되어 있음
- **When**: USER-A가 LOG_ENTRY 목록을 조회함
- **Then**: FACTORY-A의 LOG_ENTRY만 반환되고, FACTORY-B의 데이터는 **0건** 포함된다. Supabase RLS가 행 레벨에서 필터링을 적용한다.

**Scenario 2: RLS — VIEWER가 쓰기 불가**
- **Given**: VIEWER 역할의 사용자가 로그인됨
- **When**: `POST /api/v1/log-entries` (데이터 생성)를 시도함
- **Then**: 403 Forbidden이 반환되고, RLS 정책에 의해 INSERT가 차단된다.

**Scenario 3: HTTPS 전구간 암호화 확인**
- **Given**: FactoryAI API 서버가 운영 중임
- **When**: `http://` (평문) 프로토콜로 API를 호출함
- **Then**: `https://`로 301/308 리다이렉트되거나, 연결이 거부된다. 평문 통신 허용 = 0건.

**Scenario 4: 감사 로그 전수 기록 (누락률 0%)**
- **Given**: 테스트 사용자가 LOG_ENTRY CREATE 50건 + UPDATE 30건 + DELETE 20건 = 총 100건의 DB 작업을 수행함
- **When**: AUDIT_LOG 테이블을 조회함
- **Then**: 정확히 100건의 감사 로그 레코드가 존재하며, 각 레코드에 `event_type`, `actor`, `target_table`, `timestamp` 필드가 빠짐없이 기록된다. **누락률 = 0%**.

**Scenario 5: 감사 로그 — UPDATE 시 old/new 값 기록**
- **Given**: LOG_ENTRY.status를 `PENDING` → `APPROVED`로 변경함
- **When**: 해당 AUDIT_LOG 레코드를 조회함
- **Then**: `old_value: {status: "PENDING"}`, `new_value: {status: "APPROVED"}`가 정확히 기록된다.

- **Then**: 모든 엔드포인트에서 200/201 응답이 반환된다 (ADMIN = Full Access).

**Scenario 2: OPERATOR — 로깅 API만 허용, 관리 API 차단**
- **Given**: OPERATOR 역할의 사용자가 인증됨
- **When**: 다음 API를 호출함:
  - ✅ `POST /api/v1/log-entries` (로깅 — 허용)
  - ❌ `POST /api/v1/audit-reports` (리포트 생성 — 차단)
  - ❌ `GET /api/v1/audit-logs` (감사 로그 조회 — 차단)
- **Then**: 허용된 API는 정상 응답, 차단된 API는 403 Forbidden이 반환된다.

**Scenario 3: VIEWER — 조회만 허용, 모든 쓰기 차단**
- **Given**: VIEWER 역할의 사용자가 인증됨

- **Then**: GET 요청은 허용되고, 모든 POST/PATCH/DELETE 요청은 403으로 차단된다.

**Scenario 4: CISO — 보안 관련 API만 허용**
- **Given**: CISO 역할의 사용자가 인증됨
- **When**: 다음 API를 호출함:
  - ✅ `GET /api/v1/audit-logs` (감사 로그 — 허용)
  - ✅ `GET /api/v1/security/network-status` (네트워크 상태 — 허용)
  - ❌ `POST /api/v1/log-entries` (패시브 로깅 — 차단)
  - ❌ `POST /api/v1/roi/calculate` (ROI 계산 — 차단)
- **Then**: 보안 관련 API만 허용, 운영/분석 API는 차단된다.

**Scenario 5: 미인증 요청 → 401 Unauthorized**
- **Given**: 인증 토큰 없이 API를 호출함
- **When**: 임의의 API 엔드포인트에 요청함
- **Then**: 401 Unauthorized가 반환되고, 어떤 데이터도 노출되지 않는다.

**Scenario 6: JWT 역할 위조 → 차단**

**Scenario 1: 권한 부족 접근 → 차단 + 감사 로그 + CISO 알림 ≤10초**
- **Given**: VIEWER 역할의 사용자가 인증됨
- **When**: ADMIN 전용 API `DELETE /api/v1/users/{id}`를 호출함
- **Then**: ① 즉시 403 Forbidden 반환 ② AUDIT_LOG에 `{event_type: "UNAUTHORIZED_ACCESS", actor: "viewer@factory.com", target: "DELETE /api/v1/users/123", reason: "INSUFFICIENT_ROLE"}` 기록 ③ CISO에게 ≤10초 내에 "미인가 접근 시도 감지" 알림 발송

**Scenario 2: 타 공장 데이터 접근 시도 → RLS 차단 + 알림**
- **Given**: FACTORY-A 소속 OPERATOR
- **When**: FACTORY-B의 LOG_ENTRY를 직접 ID로 조회(`GET /api/v1/log-entries/{factory-b-entry-id}`)함
- **Then**: 404 Not Found(또는 403)가 반환되고, FACTORY-B 데이터는 절대 노출되지 않으며, CISO에게 "교차 공장 접근 시도" 알림이 발송된다.

**Scenario 3: 만료 토큰 사용 → 401 + 재로그인 안내**
- **Given**: JWT 토큰이 만료된 상태
- **When**: API를 호출함
- **Then**: 401 Unauthorized가 반환되고, `message: "인증이 만료되었습니다. 다시 로그인해 주세요."` 안내가 포함된다.

**Scenario 4: 5회 연속 미인가 시도 → 강화 알림**
- **Given**: 동일 사용자가 5회 연속으로 미인가 API를 호출함
- **When**: 5번째 시도가 감지됨
- **Then**: CISO에게 일반 알림이 아닌 **긴급(CRITICAL)** 등급 알림이 발송되고, AUDIT_LOG에 `severity=CRITICAL`, `pattern=REPEATED_UNAUTHORIZED_ACCESS`가 기록된다.

**Scenario 5: 차단 시 민감 데이터 미노출**
- **Given**: 미인가 사용자가 보호 API에 접근을 시도함
- **When**: 403/401 응답을 분석함
- **Then**: 응답 본문에 DB 레코드 내용, 스택 트레이스, 인프라 정보 등 민감 데이터가 **0건** 포함된다.
