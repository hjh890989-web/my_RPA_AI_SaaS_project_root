### :dart: Summary
- **기능명**: [NFR-SEC-001] HTTPS 전구간 암호화 적용 검증 (SSL 인증서 검증)
- **목적**: FactoryAI MVP의 **전 API 엔드포인트 및 프론트엔드 접속 경로**에서 HTTPS(TLS 1.2+)가 적용되어 있는지 검증한다. 평문(HTTP) 통신이 완전히 차단되어 데이터 전송 중 도청/변조가 불가능함을 보증한다. Vercel Hobby + Supabase Free 환경에서 제공되는 자동 SSL 인증서의 유효성을 확인한다.

### :link: References (Spec & Context)
> :bulb: AI Agent & Dev Note: 작업 시작 전 아래 문서를 반드시 먼저 Read/Evaluate 할 것.
- SRS 문서: [`SRS_V_1.0.md#REQ-NF-020`](file:///c:/Antigravity_Workspace/SRS%20from%20PRD_RPA%20Saas/Tasks/2_SRS_V_1.0.md) — §5.4 보안 — HTTPS 전구간 암호화 [MVP]
- 연관 기능 테스트: TEST-E6-001 (RLS + HTTPS + 감사 로그 통합 검증)
- 인프라: Vercel (자동 SSL), Supabase (자동 SSL)

### :white_check_mark: Task Breakdown (실행 계획)
- [ ] 전 API 엔드포인트 HTTPS 접근 확인: 19개 API 전부 `https://` 경로 정상 응답
- [ ] HTTP → HTTPS 리다이렉트 검증: `http://` 요청 시 301/308 리다이렉트 또는 연결 거부
- [ ] SSL 인증서 유효성 검증: 발급 기관, 만료일, 도메인 일치 확인
- [ ] TLS 버전 확인: TLS 1.2 이상만 허용, SSLv3/TLS 1.0/1.1 차단
- [ ] Supabase DB 연결 암호화 확인: `sslmode=require` 설정 검증
- [ ] Mixed Content 검증: 프론트엔드에서 HTTP 리소스 로드가 0건인지 확인
- [ ] 자동화 스크립트: `npm run security:https` 명령으로 전체 검증 실행

### :test_tube: Acceptance Criteria (BDD/GWT)

**Scenario 1: 전 API 엔드포인트 HTTPS 정상 접근**
- **Given**: FactoryAI 프로덕션 URL이 설정됨
- **When**: 19개 전체 API 엔드포인트에 `https://` 프로토콜로 요청함
- **Then**: 모든 엔드포인트가 정상 응답(2xx/4xx — 인증 필요 포함)하고, SSL 핸드셰이크가 성공함.

**Scenario 2: HTTP 평문 요청 → 차단/리다이렉트**
- **Given**: `http://` 프로토콜로 API를 호출함
- **When**: 서버에서 요청을 처리함
- **Then**: `https://`로 301/308 리다이렉트 되거나 연결이 거부됨. **평문 통신 허용 = 0건**.

**Scenario 3: SSL 인증서 유효성**
- **Given**: FactoryAI 도메인의 SSL 인증서를 조회함
- **When**: 인증서 속성을 검사함
- **Then**: ① 발급 기관이 신뢰할 수 있는 CA ② 만료일이 현재로부터 30일 이상 ③ 도메인 일치 ④ TLS 1.2+ 사용.

**Scenario 4: Supabase DB 연결 암호화**
- **Given**: Supabase 연결 문자열이 설정됨
- **When**: DB 연결 설정을 확인함
- **Then**: `sslmode=require` 또는 `verify-full`이 설정되어 있고, 평문 DB 연결이 불가함.

**Scenario 5: Mixed Content 0건**
- **Given**: FactoryAI 프론트엔드 페이지를 로드함
- **When**: 브라우저 DevTools에서 네트워크 탭을 확인함
- **Then**: `http://`로 로드되는 리소스(이미지, 스크립트, 스타일시트)가 **0건**.

### :gear: Technical & Non-Functional Constraints
- **프로토콜**: HTTPS(TLS 1.2+) 전구간 필수
- **인증서**: Vercel 자동 SSL (Let's Encrypt)
- **DB 연결**: Supabase SSL 필수 (`sslmode=require`)
- **차단**: SSLv3, TLS 1.0, TLS 1.1 완전 차단
- **테스트 도구**: Jest + supertest (API), Playwright (Mixed Content), `openssl s_client` (인증서)
- **테스트 위치**: `src/__tests__/nfr/sec/nfr-sec-001-https-enforcement.test.ts`
- **실행 명령**: `npm run security:https`

- [ ] 19개 전 API 엔드포인트 HTTPS 접근이 확인되었는가?
- [ ] HTTP 평문 요청이 차단/리다이렉트되었는가?
- [ ] SSL 인증서 유효성(CA, 만료, 도메인, TLS 버전)이 검증되었는가?
- [ ] Supabase sslmode=require가 확인되었는가?
- [ ] Mixed Content 0건이 확인되었는가?

- **Then**: 정확히 **600건**의 감사 로그가 존재. **누락률 = 0.0%**.

- **Then**: `event_type`, `actor`, `target_table`, `target_id`, `timestamp` 전 필드가 NULL 없이 기록됨. UPDATE 건은 `old_value`, `new_value`가 정확히 포함됨.

- **When**: 전 작업 완료 후 AUDIT_LOG을 조회함
- **Then**: 정확히 300건의 로그가 존재. 동시성으로 인한 누락 = 0건.

### :gear: Technical & Non-Functional Constraints

### :dart: Summary
- **기능명**: [NFR-SEC-005] ERP_CONNECTION.connection_string 암호화 저장 검증
- **목적**: ERP 연결 정보(`connection_string`)가 데이터베이스에 **평문이 아닌 암호화된 상태**로 저장되는지 검증한다. 고객의 ERP 접속 정보가 DB 탈취 시에도 안전하게 보호되어야 하며, 이는 고객 신뢰와 보안 심의 통과의 필수 요건이다.

### :link: References (Spec & Context)

- 선행 구현: [`3.0_Full TASKS list.md#DB-011`](file:///c:/Antigravity_Workspace/SRS%20from%20PRD_RPA%20Saas/Tasks/3.0_Full%20TASKS%20list.md) — ERP_CONNECTION 스키마 (connection_string 암호화 저장)

### :white_check_mark: Task Breakdown (실행 계획)
- [ ] 암호화 저장 검증: DB에 저장된 connection_string이 평문이 아님을 확인
- [ ] 암호화 알고리즘 확인: AES-256-GCM 또는 동등 이상 강도
- [ ] 복호화 정상 동작: 애플리케이션 레벨에서 복호화하여 원본 연결 문자열 사용 가능
- [ ] 암호화 키 관리: 환경변수(`ENCRYPTION_KEY`)에 키 저장, 코드 하드코딩 0건
- [ ] 평문 저장 시도 차단: 암호화 미적용 상태에서 저장 시도 시 에러 발생
- [ ] DB 직접 조회 시 불가독: Supabase Dashboard에서 직접 조회 시 암호화된 값만 표시

### :test_tube: Acceptance Criteria (BDD/GWT)

**Scenario 1: connection_string 저장 → 암호화 확인**
- **Given**: ERP 연결 정보 `host=erp.factory.com;port=5432;user=admin;pass=secret123`이 등록됨
- **When**: DB에서 해당 레코드의 `connection_string` 컬럼을 직접 조회함 (SQL)
- **Then**: 원본 텍스트가 아닌 **암호화된 바이너리/Base64 문자열**이 반환됨. `secret123`이 평문으로 존재하지 않음.

**Scenario 2: 애플리케이션 복호화 → 정상 연결**
- **Given**: 암호화 저장된 connection_string이 존재함
- **When**: 애플리케이션이 ERP 연결을 위해 connection_string을 읽어 복호화함
- **Then**: 원본 연결 문자열이 정확히 복원되어 ERP(Mock) 연결에 사용됨.

**Scenario 3: 암호화 키 없이 복호화 시도 → 실패**
- **Given**: `ENCRYPTION_KEY` 환경변수가 제거된 상태
- **When**: connection_string 복호화를 시도함
- **Then**: 복호화가 실패하고, "암호화 키 미설정" 에러가 발생함. 연결 정보가 노출되지 않음.

**Scenario 4: 암호화 키 코드 하드코딩 0건 확인**
- **Given**: 전체 소스 코드를 검색함
- **When**: 암호화 키 패턴(예: `ENCRYPTION_KEY = "..."`, `AES_KEY = "..."`)을 grep 검색함
- **Then**: 소스 코드 내 하드코딩된 암호화 키가 **0건**.

### :gear: Technical & Non-Functional Constraints

- **키 관리**: 환경변수(`ENCRYPTION_KEY`) — 코드 하드코딩 **0건**
- **보호 대상**: ERP_CONNECTION.connection_string (민감 연결 정보)
- **DB 안전성**: DB 탈취 시에도 평문 연결 정보 접근 불가
- **테스트 도구**: Jest (단위/통합), ts-jest
