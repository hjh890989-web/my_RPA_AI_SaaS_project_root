### :test_tube: Acceptance Criteria (BDD/GWT)

**Scenario 1: 정상 Lot 병합 + PDF 생성 성공**
- **Given**: FACTORY(금속가공)의 PRODUCTION_LINE(열처리)에서 APPROVED 상태의 LOG_ENTRY 50건이 존재하고, LOT 5건으로 그룹핑됨
- **When**: 관리자(AUDITOR)가 `POST /api/v1/audit-reports` (lot_ids=[L1,L2,L3,L4,L5], regulation_type="HACCP")를 요청함
- **Then**: 5개 Lot이 `captured_at` 기준 시간순으로 병합되고, 병합 정확도 ≥99%이며, 클라이언트 브라우저에서 `@react-pdf/renderer`를 통해 규제 포맷 PDF가 생성되고, AUDIT_REPORT에 `status=PENDING`으로 저장된다.

**Scenario 2: 대규모 Lot 병합 — 성능 기준 충족**
- **Given**: 100개 Lot, 총 5,000건의 LOG_ENTRY가 존재함
- **When**: 전체 Lot을 대상으로 감사 리포트 생성을 요청함
- **Then**: Lot 병합 + PDF 렌더링이 클라이언트 브라우저에서 완료되며, 전체 소요 시간이 측정/기록된다. (NFR-PERF-003 기준 참고)

**Scenario 3: PENDING/REJECTED 데이터 제외 검증**
- **Given**: 해당 기간에 APPROVED 30건, PENDING 10건, REJECTED 5건이 존재함
- **When**: Lot 병합을 수행함
- **Then**: APPROVED 상태의 30건만 Lot에 포함되고, PENDING/REJECTED 건은 병합에서 제외된다.

**Scenario 4: PDF 필수 항목 포함 검증**
- **Given**: 정상적으로 Lot 병합이 완료됨
- **When**: PDF가 생성됨
- **Then**: PDF 내에 다음 필수 항목이 모두 포함된다: ① Lot 번호 ② 생산라인명 ③ 각 로그의 타임스탬프 ④ 상태값(온도/압력 등) ⑤ 승인자 정보 ⑥ XAI 판단 근거

**Scenario 5: 클라이언트 측 생성 보증**
- **Given**: PDF 생성 요청이 발생함
- **When**: 렌더링 프로세스를 모니터링함
- **Then**: 서버 측에서 PDF 바이너리 생성 API 호출이 발생하지 않으며, 브라우저 `@react-pdf/renderer` 또는 `window.print()`만 사용됨을 확인한다.

- **Given**: 감사 리포트 생성 대상 Lot 5건 중, Lot-003에서 "열처리 온도" 필드가 NULL이고, Lot-005에서 "최종 수량" 필드가 누락됨

- **Then**: 결측치 탐지 엔진이 2건의 누락을 감지하고(감지 정확도 ≥95%), 결측치 목록(Lot-003: 열처리 온도, Lot-005: 최종 수량)을 반환하며, 해당 라인의 OPERATOR에게 ≤30초 내에 보완 요청 알림이 발송된다. PDF 생성은 보류된다.

**Scenario 2: 결측치 없음 → 정상 진행**
- **Given**: 모든 Lot의 필수 항목이 정상적으로 채워져 있음

**Scenario 3: 다수 결측치 → 우선순위 정렬 알림**

- **When**: 결측치 목록을 생성함
- **Then**: 규제 중요도(Critical > Major > Minor) 순으로 정렬된 목록이 반환되고, 가장 Critical한 항목부터 보완 가이드가 제공된다.

**Scenario 4: 보완 완료 후 리포트 재생성**
- **Given**: 이전에 결측치로 인해 보류된 리포트가 존재하고, 담당자가 누락 데이터를 보완 완료함
- **When**: 동일한 Lot 대상으로 재생성을 요청함
- **Then**: 결측치 0건 확인 후 PDF 생성이 정상 진행된다.

- **When**: Lot 병합을 시도함
- **Then**: 역전이 감지되어 병합이 차단되고, `conflict_type: "TIMESTAMP_REVERSAL"` 충돌 목록과 함께 AUDITOR에게 수동 확인 요청 알림이 발송된다.

**Scenario 3: 수동 순서 재지정 → 정상 병합 완료**
- **Given**: Scenario 1의 충돌 상황에서 관리자가 LOG_ENTRY-A를 먼저, LOG_ENTRY-B를 나중으로 수동 지정함
- **When**: 수동 해결 후 재병합을 요청함
- **Then**: 지정된 순서로 정상 병합이 완료되고, AUDIT_LOG에 "수동 충돌 해결" 이벤트가 기록된다.

**Scenario 4: 30분 이상 Gap — 경고 표시 (비차단)**
- **Given**: 연속된 LOG_ENTRY 간 시간 간격이 30분을 초과함
- **When**: Lot 병합 시 해당 Gap이 감지됨
- **Then**: 경고(`warning: "30분 이상 기록 공백 감지"`)는 표시하되, 병합을 차단하지는 않는다 (중복/역전과는 다른 처리).
