# MVP 통합 개발 실행 계획서
## 가칭: 단일 장애점 파괴 및 현장 무입력 Traceability AI 브릿지

> **문서 목적**: 고객이 해결하고자 하는 근본적인 '과제(Job-to-be-Done)'에서 출발하여 **제안 가치(Value) → MVP 핵심 기능(Feature) → 기술적 달성 목표(OKR) → 아키텍처 → 실행 로드맵 → 리스크 관리**까지를 하나의 흐름으로 통합한 실행 계획서입니다.
> **기반 문서**: 리서치 브리핑(1~6), 가치사슬 및 KSF(7~8), 문제정의(9), TAM-SAM-SOM 및 세분화(10~17), 고객여정 및 기회분석(18~27), JTBD 및 VP/PRD(29~31) 전체 종합.

## 1. 개발 핵심 철학과 리소스 제어 원칙 (Philosophy)

이전 시장 분석과 JTBD 검증(특히 '*깐깐한 박 대표*', '*현장 반장*')에서 도출된 한계를 극복하기 위해, 개발팀은 다음의 원칙을 절대적으로 준수합니다.

1. **"알고리즘 고도화보다 현장 수용성(UX)이 1순위다."**
   - 0.1%의 AI 추론 정확도를 높이는 것보다, 장갑을 낀 작업자가 아무런 화면 터치 없이(Passive) 데이터를 남길 수 있는 **Zero-Touch UX** 최적화에 개발 리소스의 70%를 쏟는다.
2. **"사일로(Silo)를 부수지 말고, 그 위에 다리(Bridge)를 놓는다."**
   - 기존 ERP(더존 등)나 MES 시스템의 데이터베이스 구조를 엎으려 하지 않는다. 기존 시스템의 원본 데이터는 보존(Read-Only)하되, 우리는 그 사이를 연결해 원청사 감사용 PDF를 출력하는 가벼운 계층(Layer)으로만 존재한다.
3. **"Fake it till you make it (초기 Concierge 모델 적극 활용)"**
   - 완벽한 백엔드 추천 알고리즘 전, 가짜(Mock-up) 웹 시뮬레이터(CFO 맞춤형 ROI 설계기)를 먼저 릴리즈하여 계약(바우처)을 따내는 것이 진짜 소프트웨어 구현보다 앞선다.

## 3. 핵심 기능(Feature) 상세 및 기술적 달성 목표 (OKR)

전체 E1~E5 중 MVP에 반드시 포함되어야 하는 최상위 우선순위 기능(P1, P2)에 대한 세부 기능 트리와, 각 Epic을 성공시키기 위한 정량적 달성 목표(OKR)를 통합합니다.

### 🥇 Priority 1 (P1): 제품 본연의 생존과 핵심 가치 창출 (Must Have)

---

#### [Epic 1] 무입력 패시브 로깅 엔진 (Zero-Touch Logging)

> 🎯 **Objective**: 현장 작업자 저항 소멸을 위한 "패시브 로깅 인프라" 구축 — 현장의 무입력/비협조 문제를 기술로 우회하여 공정 데이터를 100% 디지털화합니다.

**세부 기능 (Features)**
*   **F1.1 노이즈 캔슬레이션 STT 커맨더**: 80dB 이상의 현장 기계음 속에서도 작업자의 지시어("금형 교체", "생산 완료")만 트리거하여 텍스트로 치환하는 모듈. (Whisper API 튜닝)
*   **F1.2 Vision 상태 캡처기**: 모바일 카메라로 공정 완성품/바코드/계기판을 찍으면 LLM이 상태 값을 파싱하여 시스템에 밀어 넣는 기능.
*   **F1.3 로그 롤백/수정 관리자 웹**: AI가 오인식한 데이터를 관리자(반장)가 퇴근 전 한 화면에서 확인하고 일괄 수정(Approve/Reject)할 수 있는 웹 뷰어.

**핵심 결과 지표 (Key Results)**
| KR | 정량 목표 |
| :--- | :--- |
| KR 1.1 | 공장 소음(80dB 이상) 환경 내 음성 명령(STT) 핵심 단어 인식률 **85% 이상 확보** |
| KR 1.2 | Vision AI를 통한 설비 가동 상태 전환 하루 기준 오인식률 **5% 미만** |
| KR 1.3 | AI 오인식 시 원클릭 수정(Rollback) 예외 처리 UI **100% 구현** |

---

#### [Epic 2] 자동화된 추적성(Traceability) 컴플라이언스 봇

> 🎯 **Objective**: 영업 직결형 무기, "원클릭 Traceability 감사 리포터" — 수일이 걸리는 삼성/현대차 납품 증빙 엑셀 작업을 10분 내로 단축하여 실무진의 야근 고통을 제거합니다.

**세부 기능 (Features)**
*   **F2.1 파편화 로트(Lot) 머지 로직**: 수집된 로깅 데이터와 기존 ERP 재고 데이터를 로트 번호 기준으로 시간순 병합하는 백엔드 엔진.
*   **F2.2 템플릿 매핑 PDF 제너레이터**: 삼성전자, 현대차, EU CBAM 등 주요 규제 포맷에 맞게 테이블 뷰를 재배치하고 워터마크가 찍힌 무결성 PDF로 다운로드 및 이메일 전송하는 기능.

**핵심 결과 지표 (Key Results)**
| KR | 정량 목표 |
| :--- | :--- |
| KR 2.1 | 국내 벤더사 주요 납품 실사 포맷(PDF/Excel) 템플릿 **최소 3종 완벽 매핑** |
| KR 2.2 | 특정 로트(Lot) 번호 조회 시 데이터 병합 및 리포트 생성 속도 **10초 이내** |
| KR 2.3 | 데이터 누락 구간 발생 시 즉각 경고 로그 생성 (데이터 조작 의심 원천 차단) |

---

#### [Epic 3] 비파괴형 레거시(ERP) 브릿지

> 🎯 **Objective**: 기존 생태계 존중형 "레거시 API 브릿지" — 기존 IT 인프라와의 충돌 책임을 최소화하며 고객의 도입 공포(도입 후 시스템 멈춤)를 헷지합니다.

**세부 기능 (Features)**
*   **F3.1 Read-Only DB 커넥터**: 특정 테이블만 읽어올 수 있도록 하여 고객사 IT 담당자의 정보유출/DB손상 핑계를 원천 차단하는 플러그인.
*   **F3.2 Low-Tech 엑셀 Batch 업/다운로더**: API 개방을 절대 불허하는 극보안 타겟사를 위해, 기존 ERP 엑셀 덤프를 드래그 앤 드롭하면 우리 시스템 포맷에 자동 파싱되는 대체 기능.

**핵심 결과 지표 (Key Results)**
| KR | 정량 목표 |
| :--- | :--- |
| KR 3.1 | 국내 리딩 ERP(더존비즈온, 영림원) 규격 Open API 연동 모듈 또는 엑셀 Batch Import/Export 파이프라인 **MVP 내 구축 완료** |
| KR 3.2 | 고객사 레거시 망에 영향 없는 독립 하이브리드 클라우드 데이터 캐싱 아키텍처 **수립 완료** |

## 5. 기술 스택 및 아키텍처 초안 (Draft Tech Stack)

시간이 생명인 MVP 검증 단계를 위해 가장 유연하고 확장 가능한 스택을 채택합니다.

* **Front-End (현장 및 관리자 UX)**
  * **Framework**: React / Next.js (관리자용 대시보드 및 리포트 뷰어) + PWA (현장 작업자용 가벼운 모바일 앱 대체)
  * **UI/UX**: 키보드 입력 창 최소화, 음성 발화 버튼과 카메라 엑세스 위주의 극단적 심플 UI.
* **Back-End & Data Pipeline**
  * **Server**: Node.js 또는 Python FastAPI (비동기 처리 및 비정형 센서 데이터 고속 Ingestion)
  * **Database**: PostgreSQL (추적성 정형 데이터) + Vector DB (LLM 활용을 위한 비정형 공정 지식 및 매뉴얼 저장용)
* **AI & Machine Learning Layer**
  * **NLP/문맥이해**: OpenAI GPT-4o API (또는 Claude 3.5 Sonnet)를 호출하여 파편화된 현장 음성 및 텍스트를 정형 JSON 규격으로 파싱 (초기 자체 모델 학습 배제).
  * **로깅 센싱**: Whisper API(고도화된 STT 파이프라인 연계), 클라우드 기반 경량 Vision API.
* **Infra & Security**
  * **Cloud**: AWS 퍼블릭 클라우드 단기 활용 구조. 
  * *보안 주의*: 향후 CISO 반대에 부딪히는 기업은 AWS Outposts나 전용 VPC 망분리 구조로 업셀링(Upsell) 전략 세팅.

## 6. MVP 개발 마일스톤 100일 로드맵 (Scope Constraint)

경쟁사(대기업 SI, 기존 MES 벤더)가 무거운 통합 시스템 구축에 1년을 쓰는 동안, 우리는 **12주 만에 가장 뾰족한 문제를 해결하는 데모**를 시장에 던집니다.

| 단계 | 기간 | 주요 개발 내역 (마일스톤) | 성공 검증 타겟 |

## 7. Risk & Mitigation (개발 상의 잠재적 리스크와 헤징)

*   **리스크 1: 레거시(ERP/MES) 폐쇄성으로 인한 API 연동 실패**
    *   *헤징 방안*: 초기 MVP 단계에서는 무리한 실시간 DB Direct Access를 포기하고, 일 1회 관리자가 수동 엑셀 덤프를 추출해 드래그&드롭하면 우리 시스템이 읽어가는 **업로드/다운로드 브릿지 방식**(Low-Tech)으로 전향하여 개발 지연 방어.
*   **리스크 2: 공장 소음으로 인한 LLM/STT 환각(Hallucination)**
    *   *헤징 방안*: 자연어 완전 이해(Zero-shot)를 기대하지 않고, 현장에서 쓰는 용어(사투리, 모호한 부품명)를 Mapping Dictionary화여 **"가드레일 프롬프트"**를 단단히 세움. 인식 실패 시 강제로 수동 확인 알림을 전송하는 로직 추가.
